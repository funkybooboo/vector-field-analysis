\section{Results}

This section presents the findings of our comparative performance analysis across the various parallel implementations.


\subsection{Experimental Setup}

All performance evaluations were conducted on the Kingspeak cluster at the Center for High Performance Computing (CHPC) using four compute nodes.
The Kingspeak cluster consists of older but capable hardware, featuring Intel Xeon processors (Sandy Bridge, Ivy Bridge, Haswell, and Broadwell architectures) with 16 to 32 cores per node and an interconnect primarily based on Mellanox FDR InfiniBand.


The benchmarks involved two primary 1024 $\times$ 1024 vector field simulations: a \textit{vortex} field and a \textit{source grid divergent} field.
Each test was measured against a single static state of these fields.
To ensure statistical significance, the analysis for each implementation was repeated five times for both simulations, and the resulting metrics were averaged.
Furthermore, the correctness of each parallel implementation was validated by comparing its output against the sequential reference implementation; in all cases, the results achieved bitwise parity with the serial output, confirming identical results across all implementations.


\subsection{Shared-Memory Threading Comparison}

This study compares the performance of POSIX threads (Pthreads) against OpenMP across varying thread counts to evaluate threading overhead and efficiency.

\begin{table}[htbp]

	\label{tab:threading_comparison}
	\begin{tabular}{|l|c|c|}
		\hline
		\textbf{\# Threads} & \textbf{POSIX Threads Runtime (ms)} & \textbf{OpenMP Runtime (ms)} \\ \hline
		1 (Serial)          & 1706.96                             & 1706.96                      \\ \hline
		2                   & 1670.89                             & 1646.21                      \\ \hline
		4                   & 1701.03                             & 1645.43                      \\ \hline
		8                   & 1721.48                             & 1638.58                      \\ \hline
		16                  &                                     &                              \\ \hline
	\end{tabular}
\end{table}


The results indicate that both Pthreads and OpenMP implementations scale effectively with increasing core counts, showing consistent performance improvements at each stage.
However, OpenMP consistently achieved slightly lower runtimes than Pthreads across all configurations.
This advantage was particularly evident at the 4-thread count, where OpenMP's internal load balancing mechanisms better managed the non-uniform computational load inherent in streamline tracing compared to the manual Pthreads implementation.

\subsection{GPU Block Size Analysis}

\begin{table}[htbp]
	\centering
	\caption{GPU Performance vs. Block Size}
	\label{tab:gpu_block_size}
	\begin{tabular}{|l|r|r|}
		\hline
		\textbf{Block Size} & \textbf{Runtime (ms)} & \textbf{Speedup} \\ \hline
		$128 \times 128 $   &                       &                  \\ \hline
		$256 \times 256 $   &                       &                  \\ \hline
		$512 \times 512 $   &                       &                  \\ \hline
	\end{tabular}
\end{table}

The GPU performance analysis demonstrates that increasing the block size from 128 to 256 threads provides a substantial performance boost, reducing runtime from 549.
4 ms to 471.2 ms.
This improvement suggests that a block size of 256 better utilizes the GPU's streaming multiprocessors and improves latency hiding.
Moving to a block size of 512 threads yielded a much smaller improvement, reaching 468.6 ms, indicating that the system is approaching a point of diminishing returns where occupancy is already high enough to saturate the available hardware resources.


\subsection{Distributed CPU vs. Distributed GPU Scaling Study}

\begin{table}[htbp]
	\centering
	\caption{Distributed Scaling Comparison (2--4 Nodes)}
	\label{tab:distributed_scaling}
	\begin{tabular}{|l|c|c|}
		\hline
		\textbf{Nodes} & \textbf{Distributed CPU Runtime (s)} & \textbf{Distributed GPU Runtime (s)} \\ \hline
		2 Nodes        &                                      &                                      \\ \hline
		4 Nodes        & 2.73                                 &                                      \\ \hline
	\end{tabular}
\end{table}

The distributed-memory implementation underperformed in comparison to its shared memory counterparts.
While the neighbor calculation phase was able to be scaled effectively due to the isolated calculations, the parallel path merging phase relies heavily on accessing data from other parts of the vector field when working across the boundaries.
This high overhead stems from the requirement to reconcile streamline connectivity across process boundaries.
Because each node only possesses local knowledge of its assigned sub-grid, resolving global paths requires iterative synchronization to converge the distributed Disjoint Set Union (DSU) structure.
As the node count increases, the latency associated with these global reductions and data transfers becomes a dominant factor, limiting the overall scalability of both the distributed CPU and hybrid MPI+CUDA implementations.
